# Shadow House v1

Первый бесплатный модуль системы **«Дом»**: чатовый сценарий **«Тень дома»** с FastAPI backend, Next.js frontend и PostgreSQL.

## Что делает проект сейчас

- пользователь проходит почти весь модуль без регистрации;
- имя спрашивается в самом начале сценария;
- перед выдачей «Паспорта тени» открывается мягкий шаг сохранения результата;
- на этом шаге имя уже подставлено из начала, но его можно поправить;
- затем пользователь вводит телефон и подтверждает его кодом;
- после подтверждения сохраняются пользователь, телефон, сессия и итоговый результат;
- OpenAI вызывается только через backend, без ключей во frontend;
- без OpenAI проект работает через fallback-LLM и не ломает сценарный flow.

## Пользовательский flow

1. Пользователь открывает ссылку и начинает сессию.
2. КЭТ спрашивает имя.
3. КЭТ спрашивает возраст.
4. Пользователь проходит весь модуль «Тень дома».
5. Вместо мгновенной выдачи паспорта появляется шаг сохранения результата.
6. На этом шаге:
   - имя уже подставлено из раннего ответа;
   - имя можно поправить;
   - телефон спрашивается только здесь;
   - затем отправляется код подтверждения.
7. После успешного подтверждения кода:
   - создаётся или обновляется пользователь;
   - текущая анонимная сессия привязывается к пользователю;
   - итоговый «Паспорт тени» разблокируется и сохраняется.

## Где лежат данные

### `users`

Хранит подтверждённого пользователя:

- `id`
- `display_name`
- `phone_number`
- `phone_verified`
- `phone_verified_at`

### `user_sessions`

Хранит саму сессию модуля и её итог:

- `id` — session id
- `user_id` — связь с подтверждённым пользователем
- `contact_name` — имя для этой сессии / результата
- `phone_number`
- `phone_verified_at`
- `state` — прогресс, ответы, промежуточные данные и pending result до подтверждения
- `result` — финальный сохранённый паспорт
- `result_released_at`
- `access_token_hash` — хеш токена доступа к результату

### `phone_verifications`

Хранит OTP-попытки:

- `session_id`
- `phone_number`
- `display_name`
- `code_hash`
- `provider`
- `status`
- `expires_at`
- `attempts_count`
- `max_attempts`
- `verified_at`

## OTP / подтверждение телефона

Сейчас реализована backend-архитектура подтверждения телефона:

- запрос кода;
- хранение кода только в виде hash;
- TTL кода;
- лимит попыток;
- привязка к `session_id` и номеру телефона;
- разблокировка результата только после успешной верификации;
- отдельный токен доступа к сохранённому результату.

### Dev mode

Для локальной проверки по умолчанию включён:

```bash
OTP_PROVIDER=dev
OTP_DEV_MODE=true
```

В этом режиме backend:

- не отправляет реальную SMS через внешний сервис;
- логирует код в backend;
- возвращает `dev_code` в ответе `request-code`, чтобы можно было пройти flow локально.

Это **только dev-flow**. Это не означает, что SMS уже готова для production.

### Production

Для production нужно подключить реальный SMS-провайдер через backend.

Сейчас для него уже есть точка расширения через `OTP_PROVIDER`, но внешний провайдер и его ключи ещё не подключены.

## OpenAI и fallback

По умолчанию:

```bash
OPENAI_ENABLED=false
```

Тогда backend использует fallback-генератор и сценарий остаётся полностью рабочим.

Если нужен OpenAI:

```bash
OPENAI_ENABLED=true
OPENAI_API_KEY=ваш_ключ
OPENAI_MODEL=gpt-5.2
```

Важно:

- OpenAI key хранится только на backend;
- frontend знает только `NEXT_PUBLIC_API_BASE_URL`;
- `NEXT_PUBLIC_OPENAI_*` переменные не используются.

## Локальный запуск

1. Скопируйте `.env.example` в `.env`.
2. При необходимости впишите `OPENAI_API_KEY`.
3. Оставьте dev OTP включённым для локальной проверки:

```bash
OTP_PROVIDER=dev
OTP_DEV_MODE=true
```

4. Запустите проект:

```bash
docker compose up --build
```

После запуска:

- frontend: [http://localhost:3000](http://localhost:3000)
- backend docs: [http://localhost:8000/docs](http://localhost:8000/docs)

## Как локально проверить новый flow

1. Откройте сайт.
2. Пройдите сценарий почти до конца.
3. Убедитесь, что перед «Паспортом тени» появился шаг сохранения результата.
4. Проверьте, что имя уже подставлено из начала сценария.
5. При желании поправьте имя.
6. Введите телефон.
7. Нажмите отправку кода.
8. Используйте `dev_code`, который backend вернёт в dev-режиме.
9. Подтвердите код.
10. Убедитесь, что после этого открылся «Паспорт тени».

- `POST /api/v1/sessions`
- `GET /api/v1/sessions/{session_id}`
- `POST /api/v1/sessions/{session_id}/messages`
- `POST /api/v1/sessions/{session_id}/contact/request-code`
- `POST /api/v1/sessions/{session_id}/contact/verify-code`
- `GET /api/v1/sessions/{session_id}/result`
- `GET /api/v1/analytics/overview`
- `GET /api/v1/health`

## Что реально готово, а что ещё нет

### Уже готово

- мобильная адаптация интерфейса;
- мягкий шаг имени / телефона перед результатом;
- backend OTP-архитектура;
- dev-flow для локальной проверки кода;
- сохранение пользователя, телефона, сессии и результата;
- защита OpenAI-секрета от frontend;
- базовая защита результата через access token.

### Ещё не доведено до production

- реальная отправка SMS через внешний провайдер;
- полноценная авторизация пользователя между устройствами;
- отдельный кабинет с историей модулей;
- полноценные миграции БД через Alembic;
- расширенный security-hardening и rate limiting на уровне reverse proxy / WAF.
